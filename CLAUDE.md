# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

A Godot 4.7 (Forward+ renderer, Jolt physics, D3D12 on Windows) third-person immersive-sim/RPG sandbox, currently greenfield/prototype stage. No README, no test suite, no CI — this file is the only project-level documentation.

## Commands

Godot executable (Steam install, not on PATH — this machine has multiple Steam libraries across drives; check `HKCU:\Software\Valve\Steam` / `libraryfolders.vdf` if this path stops working after a Steam library change):
```
"E:\SteamLibrary\steamapps\common\Godot Engine\godot.windows.opt.tools.64.exe"
```

Check the whole project for script/scene parse errors (fast, no window):
```
godot.exe --headless --path <project> --check-only --quit
```
Note: on a machine/checkout where `.godot/` hasn't been generated yet (e.g. after adding new `class_name` scripts), `--check-only` alone can report false "Could not find type X" errors because the global class cache isn't built. Fix by first running:
```
godot.exe --headless --editor --path <project> --quit-after 20
```
then re-run `--check-only`.

Run the actual game (not the editor) directly into a scene:
```
godot.exe --path <project> res://Scenes/level.tscn
```
Running `godot.exe --path <project>` with no scene argument opens the **editor**, not the game — the window title says "... - Godot Engine" when that happens.

There is no build step, linter, or automated test suite in this project.

## Architecture

### Plugin-style ability system (`Scripts/Magic/`)
`SpellCaster` (a `Node` under the player's `Camera3D`) iterates its own `Spell`-derived children every physics frame and drives each through `start_cast()` → `hold_cast(delta)` (repeatedly while held) → `end_cast()`, keyed off each spell's own `trigger_action` input action. Adding a new ability means adding a new `Spell` subclass as a child scene under the camera — `SpellCaster` itself never needs to change. `instant` spells (`@export var instant`) skip the hold/release phase and fire once on press.

Spells affect the world via **duck typing, not shared base classes**: e.g. `TelekinesisPushSpell`/`TelekinesisGrabSpell` call `apply_impulse(impulse, position)` on anything that has that method (checked via `has_method`), and target the `"grabbable"` group for grab specifically. `SpiderWalker` implements `apply_impulse()` itself (folding the impulse into `velocity` since it's a `CharacterBody3D`, not a `RigidBody3D`) purely so spells can affect it without knowing its concrete type. **Follow this convention** for anything new that should be pushable/grabbable — implement `apply_impulse()`, don't add a shared interface class.

### Enemies (`Enemies/`)
`SpiderWalker` is fully self-contained and procedural: drop the scene anywhere in a level with zero required configuration and it wanders on its own (centered on wherever it was placed), optionally chasing the `"player"` group if `chase_player` is set. All leg motion is IK-driven at runtime by `SpiderGait`/`SpiderSkeletonBuilder` (`spider_gait.gd`, `spider_skeleton_builder.gd`) — there are no animation clips. Health/damage is handled by the `Components/` system below; knockback (`apply_impulse()`) remains separate from damage.

`EnemySpawner` (`Enemies/Spawner/enemy_spawner.tscn`) is a drop-in, destructible spawner: a `StaticBody3D` (mesh + collision + `DamageReceiver` + floating `HealthBar3D`) that instances its exported `enemy_scene` (defaults to `SpiderWalker`) on a timer, keeping up to `max_alive` alive at once and refilling a slot as they die (`max_total` 0 = infinite; also `spawn_interval`/`first_spawn_delay`/`spawn_radius`/`activation_radius` for proximity gating/`auto_start`). Spawned enemies are parented as **siblings** (`get_parent().add_child`) — set to their spawn position *before* `add_child` since self-contained enemies read `global_position` in `_ready` — so destroying the spawner (its `DamageReceiver` FREE_PARENTs it) only stops future spawns; existing enemies live on. Extend per-enemy setup by overriding `_configure_spawned(enemy)`; spawners join group `"enemy_spawner"`.

### Progression (`Scripts/Progression/player_progression.gd`)
`ProgressionSystem` is registered as the **`PlayerProgression` autoload** (listed before `InventoryUI` so it exists when the screen's `_ready` runs). It holds level, XP, unspent attribute points, unspent skill points, and the attribute values (STR/DEX/INT/VIT/LUK). Being an autoload it **persists across scene reloads**, so the player-death reload (`DamageReceiver` RELOAD_SCENE) doesn't wipe progress. Award XP from anywhere via `PlayerProgression.add_experience(x)` — the reusable entry point meant to be called by quests/objectives/social outcomes later, not just kills. XP-from-kills is wired through `DamageReceiver.xp_reward` (0 = grants none; the spider sets 25): on death the receiver looks up `/root/PlayerProgression` defensively (no hard dependency) and awards. XP curve: `round(base_xp * level^curve_exponent)`, all exported/tunable; each level grants `attribute_points_per_level` (3) + `skill_points_per_level` (1). **Attributes and skill points are COSMETIC right now** — spending attribute points changes the displayed number but nothing gameplay-facing, and skill points are just a tracked currency for a future ability-unlock system. The two hook methods `_apply_level_up()` / `_apply_attributes()` (both no-ops, grep "COSMETIC") are where to wire real effects (max HP from VIT, spell damage from INT, heal-on-level, etc.) when stats should start mattering. The character screen (`inventory_screen.gd`) binds to the progression signals and **builds the attribute rows dynamically** from `ProgressionSystem.ATTRIBUTES` (name/value/`+`-button per row) rather than hand-placing them — the `+` buttons and points readout only appear once there are points to spend. Level-ups also fire a fading toast on the HUD (below).

### HUD & player resources (`Scenes/UI/hud.tscn`, `Scripts/Player/player_state.gd`)
The **`HUD` autoload** (a `CanvasLayer`, layer 5) is the always-on gameplay overlay: health/mana/stamina bars (bottom-left), an XP bar + level readout (bottom strip), a gold counter (top-right), a mission-objective panel (top-left, retask via `set_objective(title, text)`), and the centered level-up toast. It's display-only — every node is `mouse_filter = IGNORE` so it never eats casting clicks. It binds to three sources: the player's `DamageReceiver` for health (found via group `"player_health"`, **re-acquired on scene reload** by listening to `get_tree().node_added` since the death-reload recreates the player), `PlayerProgression` for XP/level + the toast, and `PlayerState` for gold/mana/stamina. The inventory/character screen hides the HUD while open via `set_hud_visible(false)` (group `"hud"` lookup) so it doesn't show behind the backdrop.
**`PlayerStateSystem`** is the **`PlayerState` autoload** holding gold plus the mana/stamina pools (current/max + regen + `spend_*`/`restore_*`/`add_gold`/`spend_gold`, with `*_changed` signals). PLACEHOLDER: nothing drains mana/stamina yet (spells/sprint are free) so they sit full; the API exists for spell-cost/sprint-drain to hook in later, and gold is seeded (50) with no source/sink beyond the API until the economy/bribery systems arrive. Health is NOT here — it stays on the player's `DamageReceiver` (per-body); mana/stamina/gold are account-level in this singleton, an intentional split.

### Damage components (`Components/`)
`DamageReceiver` (health + `damaged`/`health_changed`/`died` signals + `heal()` + per-instance `death_behavior`: none / free parent / reload scene) and `DamageDealer` (damage amount + per-target cooldown) are child-node components, instanced as direct children of a body's scene root and configured per instance in the `.tscn`. `health_changed(current, max)` fires on every change and once on `_ready`, so UI (HUD, character sheet) can bind to it as the single source of truth; the player's receiver is in group `"player_health"` for discovery. The convention: the *owner* detects contact (projectile impact callback, spider slide collisions) and calls `dealer.try_deal(body)`; the dealer finds the body's receiver via `DamageReceiver.find_in(body)` (direct-children scan, so receivers must sit at the body root). Both are optional everywhere — scripts look them up with `get_node_or_null` and skip damage if absent, so a scene without components still works. Current wiring: player receiver (100 HP, reload scene on death), spider dealer (touch, 10 dmg / 1s cooldown) + receiver (50 HP), barrel receiver (20 HP), Fireball/Ice Bolt projectile dealers (25/15 dmg). There is still no loot-drop system; death is queue_free (or scene reload for the player).

### Player (`Scenes/brackeys-proto-controller-main/`)
The player is the unmodified third-party Brackeys `ProtoController` (`CharacterBody3D`, group `"player"`), configured via exported input-action-name strings rather than hardcoded actions. It self-manages mouse capture (`capture_mouse()`/`release_mouse()`) — left-click captures, Escape releases — polled every `_unhandled_input`, not event-filtered. Anything that opens a full-screen UI must consume input at the `Control` layer (`mouse_filter = STOP` on a full-rect backdrop) so clicks don't fall through to `_unhandled_input` and re-trigger capture.

### UI (`Scripts/UI/`, `Scenes/UI/`)
The inventory/character screen is the first UI in the project and sets the conventions for future screens:
- Registered as an **autoload singleton** (`InventoryUI` in `project.godot`) rather than a node manually placed in every level, so it's available everywhere and isn't tied to `level.tscn`.
- Toggled by the `inventory` input action (Tab, already mapped in `project.godot` before this system existed). Opening sets `get_tree().paused = true` and `Input.mouse_mode = VISIBLE`; the screen node and everything under it must have `process_mode = PROCESS_MODE_ALWAYS` (set on the root `CanvasLayer`) or it stops receiving input the moment the tree pauses.
- `InventorySlot` (`Scenes/UI/inventory_slot.tscn`) is a single dual-purpose component used for both backpack grid cells and paper-doll equipment sockets (`is_equipment_slot` + `accepted_equip_slot` toggle the behavior) — not two separate slot classes. It owns its own drag-and-drop (`_get_drag_data`/`_can_drop_data`/`_drop_data`) and double-click quick-transfer, and notifies the screen via `get_tree().get_first_node_in_group("inventory_screen")` rather than a direct reference.
- `InventoryItem` (`Scripts/UI/item.gd`) is a plain `Resource` data shape with no backing item/save system yet. `MockItemDatabase` (`Scripts/UI/mock_item_database.gd`) is placeholder demo data standing in for that system — replace it, don't extend it, once real loot/persistence exists.
- Shared look-and-feel lives in one `Theme` resource (`Scenes/UI/inventory_theme.tres`, dark panels + brass borders) applied at the screen root so new UI should pull from it rather than re-declaring styleboxes per-scene. This file gets hand-edited/re-saved by the Godot editor directly (not just through Claude), so re-read it before assuming its contents rather than trusting an earlier diff.

The **journal screen** (`Scripts/UI/journal_screen.gd`, `JournalUI` autoload, toggled by `journal`/J) is the second full-screen menu and copies all of the above conventions (autoload, backdrop, theme, `process_mode = ALWAYS`, pause + mouse release, HUD-hide). It's a **read-only log**: a left `ItemList` of entry titles + a right `RichTextLabel` for the selected entry's body (`JournalEntry` data shape — same mock-data pattern as `MockItemDatabase`/`MockSpellbook`). `add_entry(JournalEntry)` appends one at runtime; `set_entries(Array[JournalEntry])` replaces the whole log. `MockJournal` is now only a **generic fallback** shown when a level provides no objectives of its own.

**Per-level objectives** (`Scripts/Level/level_objectives.gd`): each level scene contains a `LevelObjectives` node (a plain `Node`, edited in the inspector — the single place to change a level's goals). On load it (deferred) pushes its `objective_title`/`objective_text` to the HUD objective panel (`group "hud"` → `set_objective`) and its entries to the journal (`group "journal_screen"` → `set_entries`), **replacing** the previous level's — so `level.tscn` shows "Remove the Foreman" + dock lore and `level_2.tscn` shows "Breach the Keep". `objective_in_journal` (default true) auto-adds the objective as the first journal entry so it's written once; `journal_entries: Array[JournalEntry]` holds the level's extra lore/clues (authored as `type="Resource"` sub-resources with `script` = `journal_entry.gd`; the typed-array serialises as `Array[JournalEntry]([SubResource(...)])`). A level without a `LevelObjectives` node falls back to the HUD's neutral default + `MockJournal`. **Full-screen menus are mutually exclusive** via the `"menu_screen"` group: both `InventoryScreen` and `JournalScreen` join it and call `_close_other_menus()` in `open()`, so opening one closes the other — without this, closing a second-opened menu would unpause the tree and recapture the mouse while the first was still visible.

### Level scene (`Scenes/level.tscn`)
Mostly hundreds of hand-placed `Floor`/`Wall` tile instances plus a few set-piece nodes (`ProtoController`, `SpiderWalker`, pirate ship, barrels). When editing this file directly as text, be careful with `unique_id` node attributes and don't try to read/edit the whole floor-tile block — it's bulk, not structure.

### Input map (`project.godot`)
Actions are pre-declared even when unused by any script yet (e.g. `interact` on F was unused until recently, `inventory` on Tab similarly) — check `[input]` in `project.godot` before assuming an action needs to be created for a new feature.

### Magic loadout system (Fireball/Ice Bolt, spellbook tab)
`SpellCaster.equip_spell(trigger_action, spell_scene)` / `unequip_slot(trigger_action)` dynamically add/remove a `Spell`-derived scene as a child so a hotbar slot's binding can change at runtime — used by the inventory's Spellbook tab. Every equippable ability (`TelekinesisGrab/Push`, `FireballSpell`, `IceboltSpell`) is authored as its own standalone `.tscn` under `Scripts/Magic/` (not inline nodes in `proto_controller.tscn`) specifically so it can be `instance=`'d both as the default loadout in `proto_controller.tscn` *and* dynamically via `equip_spell()`. `ProjectileSpell`/`Projectile` are generic (spawn-and-fly-and-impact) scripts parametrized per spell via scene config (mesh/particle colors, speed, impact force) rather than duplicated per spell — adding a third projectile spell means authoring a new `Projectile` scene, not new code. `SpellDefinition`/`MockSpellbook` mirror the `InventoryItem`/`MockItemDatabase` mock-data pattern.

### `inventory_screen.tscn` convention: `$path` for instanced nodes, `%unique_name` for plain nodes
`InventorySlot`/`SpellSlot`/`ItemTooltip` nodes (placed via `instance=ExtResource(...)`) are looked up by explicit `$relative/path/To/Node` (see the `_CHAR_PATH`/`_SPELL_PATH` constants in `inventory_screen.gd`), not `%unique_name`. Plain built-in node types (`Button`, `Label`, `GridContainer`, etc.) use `%unique_name` as normal. This was originally adopted because an earlier session saw a `WARNING: Parent path '...' has vanished when instantiating` cascade and attributed it to `unique_name_in_owner` on instanced nodes corrupting `PackedScene::instantiate()`. **That diagnosis turned out to be wrong** — a later occurrence of the exact same warning (2026-07-17) was empirically bisected (isolated repro scenes, stripping `unique_name_in_owner` project-wide, de-duplicating node names — none of it helped) down to a mundane stale `parent=` NodePath: when a `Layout` `VBoxContainer` was inserted as a new wrapper around `Header`/`Tabs`, every descendant of `Tabs` kept the old pre-`Layout` parent path and was never updated, so Godot correctly reported the path as not found. **If you see this warning again: check `parent=` attributes against actual node nesting first** — grep for the container's own `parent=` line and confirm every one of its descendants' paths actually contains it — before suspecting `unique_name_in_owner` or any other engine-level explanation. The `$path`/`%unique_name` split above is kept as-is since it works and isn't worth re-churning, not because the original bug theory was confirmed.

Separately: don't hand-write `uid="uid://..."` attributes when authoring new `.tscn`/`.tres` files by hand (outside the editor) — a fabricated UID isn't registered anywhere and risks colliding with one Godot generates for something else during a full reimport. Omit `uid=` and reference by `path=` only; Godot fills in a real UID the next time the file is saved through the actual editor.
