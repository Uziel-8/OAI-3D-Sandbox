
## Architecture

### Plugin-style ability system (`Scripts/Magic/`)
`SpellCaster` (a `Node` under the player's `Camera3D`) iterates its own `Spell`-derived children every physics frame and drives each through `start_cast()` â†’ `hold_cast(delta)` (repeatedly while held) â†’ `end_cast()`, keyed off each spell's own `trigger_action` input action. Adding a new ability means adding a new `Spell` subclass as a child scene under the camera â€” `SpellCaster` itself never needs to change. `instant` spells (`@export var instant`) skip the hold/release phase and fire once on press.

Spells affect the world via **duck typing, not shared base classes**: e.g. `TelekinesisPushSpell`/`TelekinesisGrabSpell` call `apply_impulse(impulse, position)` on anything that has that method (checked via `has_method`), and target the `"grabbable"` group for grab specifically. `SpiderWalker` implements `apply_impulse()` itself (folding the impulse into `velocity` since it's a `CharacterBody3D`, not a `RigidBody3D`) purely so spells can affect it without knowing its concrete type. **Follow this convention** for anything new that should be pushable/grabbable â€” implement `apply_impulse()`, don't add a shared interface class.

### Enemies (`Enemies/`)
`SpiderWalker` is fully self-contained and procedural: drop the scene anywhere in a level with zero required configuration and it wanders on its own (centered on wherever it was placed), optionally chasing the `"player"` group if `chase_player` is set. All leg motion is IK-driven at runtime by `SpiderGait`/`SpiderSkeletonBuilder` (`spider_gait.gd`, `spider_skeleton_builder.gd`) â€” there are no animation clips. Health/damage is handled by the `Components/` system below; knockback (`apply_impulse()`) remains separate from damage.

`EnemySpawner` (`Enemies/Spawner/enemy_spawner.tscn`) is a drop-in, destructible spawner: a `StaticBody3D` (mesh + collision + `DamageReceiver` + floating `HealthBar3D`) that instances its exported `enemy_scene` (defaults to `SpiderWalker`) on a timer, keeping up to `max_alive` alive at once and refilling a slot as they die (`max_total` 0 = infinite; also `spawn_interval`/`first_spawn_delay`/`spawn_radius`/`activation_radius` for proximity gating/`auto_start`). Spawned enemies are parented as **siblings** (`get_parent().add_child`) â€” set to their spawn position *before* `add_child` since self-contained enemies read `global_position` in `_ready` â€” so destroying the spawner (its `DamageReceiver` FREE_PARENTs it) only stops future spawns; existing enemies live on. Extend per-enemy setup by overriding `_configure_spawned(enemy)`; spawners join group `"enemy_spawner"`.

### Data-driven NPCs (`NPCs/`, `Scripts/NPCs/`)
NPCs are composed from **three legs**, so you can drop a bunch in and iterate the majority centrally: (1) **Godot scene inheritance** for structure/model â€” a base `NPCs/npc.tscn`, and each archetype is an *inherited scene* that swaps the model, assigns a profile, and adds only its custom bits; (2) an **`NpcProfile` resource** (`Scripts/NPCs/npc_profile.gd`, `.tres` under `Resources/NPCs/`) holding shared archetype DATA; (3) the existing drop-in **components + FSM** for behaviour (`DamageReceiver`/`Interactable`/`HealthBar3D` + a `StateMachine`). Data lives in the profile, behaviour in the states, per-NPC tweaks are exports on the instance.

`Npc` (`npc.gd`, `class_name Npc`, `CharacterBody3D`, groups `"npc"`+`"interactable"`+`"faction_<faction>"`) is the **context**, exactly like `PlayerController`: in `_ready` it copies the profile into live state + the components, and it exposes the movement *primitives* the states call (`apply_gravity`, `steer_towards`, `face_towards`, `halt`, `tick_stagger`) â€” it decides nothing itself. It implements `apply_impulse()` (folds knockback into velocity + a brief stagger) so spells push NPCs, per the duck-typing convention. **`NpcProfile` is IMMUTABLE shared data** (faction, `starting_disposition`, class, vitals, a `Movement` enum, dialogue): many NPCs share one `.tres`, so editing the archetype updates all of them â€” which means **never mutate the profile at runtime** (you'd change every NPC sharing it, and in-editor write it back to disk). Live state is seeded onto the node/components: current health on the `DamageReceiver` (pushed through in `_apply_profile` since the component's `_ready` already ran), current `disposition` on the `Npc`. Profile dialogue/prompt fill the `Interactable` **unless the scene authored its own** (scene wins), so an archetype gives defaults and a specific NPC overrides the last bit.

**Movement is behaviour â†’ FSM states** reusing the generic `StateMachine` (the "NPC reuse of StateMachine" that was flagged as a next step): `NpcStaticState`/`NpcPatrolState`/`NpcChaseState` (`Scripts/NPCs/States/`, base `NpcState` resolves its `Npc` via `get_parent().get_parent()`, not a group, since there are many NPCs). `Npc._ready` sets `state_machine.initial_state` from `profile.movement` **before** the SM's deferred first `enter()`, so it starts in the right state. Hostility (`disposition < 0`) *layers on top* of movement: a hostile Static/Patrol NPC breaks off to Chase when the player is within `chase_range`; Chase returns to the default movement past `give_up_range` (a pure `CHASE` archetype pursues relentlessly). Patrol walks `patrol_points` â€” an `Array[Node3D]` of waypoints placed **in the level**, not the profile (a shared resource can't hold per-instance world positions). **Adding a movement = a new state; adding an archetype = a new profile + inherited scene** â€” neither touches the others. Shipped archetypes: `dock_foreman.tscn` (static/friendly), `guard.tscn` (hostile â†’ chases), `patroller.tscn` (patrol), with profiles in `Resources/NPCs/`.

`dock_foreman.tscn` is now an **inherited scene of `npc.tscn`** (kept its original `uid` + root node name `DockForeman`, so `level.tscn` and the kill/interact objectives' `../../DockForeman` NodePaths still resolve); its dialogue moved into `foreman_profile.tres`. **Not yet done** (clear extension points): NPCs have no locomotion animator â€” each model just autoplays a single `Idle` clip, so a *moving* NPC slides (a small NPC animator, or reusing `PlayerAnimator`'s pattern, is the follow-up); Chase deals no contact damage yet (give the NPC a `DamageDealer` and `try_deal` in `NpcChaseState`, the "owner detects contact" convention, like SpiderWalker); the `"faction_<x>"` groups exist but nothing reads them (faction-vs-faction hostility is future). KayKit rigs face **+Z**, so facing uses `atan2(dir.x, dir.z)` like SpiderWalker â€” flip a model in its inherited scene if it ends up backwards.

### Progression & live attributes (`Scripts/Progression/player_progression.gd`)
`ProgressionSystem` is registered as the **`PlayerProgression` autoload** (listed before `InventoryUI` so it exists when the screen's `_ready` runs). It holds level, XP, unspent attribute points, unspent skill points, and the attribute values. Being an autoload it **persists across scene reloads**, so the player-death reload (`DamageReceiver` RELOAD_SCENE) doesn't wipe progress. Award XP from anywhere via `PlayerProgression.add_experience(x)` â€” the reusable entry point meant to be called by quests/objectives/social outcomes later, not just kills. XP-from-kills is wired through `DamageReceiver.xp_reward` (0 = grants none; the spider sets 25): on death the receiver looks up `/root/PlayerProgression` defensively (no hard dependency) and awards. XP curve: `round(base_xp * level^curve_exponent)`, all exported/tunable; each level grants `attribute_points_per_level` (3) + `skill_points_per_level` (1).

**Attributes are LIVE** (magic-focused set; STR and LUK were deliberately cut): four dual-purpose stats, flat linear scaling per point above `BASELINE` (10), all per-point scalars exported on the progression node (the single tuning home): **CON** max HP (+8/pt) & HP regen (+0.2/s/pt, base 0); **DEX** stamina regen (+5%/pt) â€” *its casting-speed half is deliberately deferred until a spell-cooldown system exists*; **INT** magic damage (+3%/pt) & max mana (+6/pt); **WIL** magic damage (+2%/pt) & mana regen (+5%/pt). Consumers: the player `DamageReceiver` is pushed to directly (via `node_added` on group `"player_health"` â€” *before* its `_ready` on spawn so health initialises at the boosted max; raising max later does NOT refill current health/mana), `PlayerState` recomputes off the `derived_stats_changed` signal, and `ProjectileSpell` multiplies each spawned projectile's `DamageDealer.damage` by `magic_damage_multiplier()` (fresh instance per cast, so it never compounds). Skill points are the currency spent in the **magic skill trees** (see "Magic skill trees" below) via `spend_skill_point()`/`grant_skill_points()`, kept here so the currency stays single-sourced. The character screen (`inventory_screen.gd`) binds to the progression signals and **builds the attribute rows dynamically** from `ProgressionSystem.ATTRIBUTES` (name/value/`+`-button per row) rather than hand-placing them â€” the `+` buttons and points readout only appear once there are points to spend. Level-ups also fire a fading toast on the HUD (below).

### HUD & player resources (`Scenes/UI/hud.tscn`, `Scripts/Player/player_state.gd`)
The **`HUD` autoload** (a `CanvasLayer`, layer 5) is the always-on gameplay overlay: health/mana/stamina bars (bottom-left), an XP bar + level readout (bottom strip), a gold counter (top-right), a mission-objective panel (top-left, retask via `set_objective(title, text)`), and the centered level-up toast. It's display-only â€” every node is `mouse_filter = IGNORE` so it never eats casting clicks. It binds to three sources: the player's `DamageReceiver` for health (found via group `"player_health"`, **re-acquired on scene reload** by listening to `get_tree().node_added` since the death-reload recreates the player), `PlayerProgression` for XP/level + the toast, and `PlayerState` for gold/mana/stamina. The inventory/character screen hides the HUD while open via `set_hud_visible(false)` (group `"hud"` lookup) so it doesn't show behind the backdrop.
**`PlayerStateSystem`** is the **`PlayerState` autoload** holding gold plus the mana/stamina pools (`spend_*`/`restore_*`/`add_gold`/`spend_gold`, with `*_changed` signals). **Pools are LIVE**: spells charge `mana_cost` at press time (`Spell.try_pay_cost()`, enforced by `SpellCaster` before `start_cast` â€” Fireball 15 / Ice Bolt 12 / Push 5 / Grab free, set on each spell `.tscn`, matching the numbers the spellbook tooltip already displayed), and sprint-while-moving drains stamina (`_try_pay_sprint` in `proto_controller.gd` â€” **the Brackeys controller is no longer unmodified**, this is its one local change; sprint falls back to walk speed when the pool is dry). After any spend, that pool's regen pauses briefly (`mana_regen_delay` 0.5s / `stamina_regen_delay` 1.0s) so drains bite. Caps/regen are exported bases modified by attributes (INT/WIL/DEX â€” see Progression above). Gold is seeded (50) with no source/sink beyond the API until the economy/bribery systems arrive. Health is NOT here â€” it stays on the player's `DamageReceiver` (per-body, now with an exported `health_regen`/s driven by CON for the player, usable per-instance on any receiver); mana/stamina/gold are account-level in this singleton, an intentional split.

### Damage components (`Components/`)
`DamageReceiver` (health + `damaged`/`health_changed`/`died` signals + `heal()` + per-instance `death_behavior`: none / free parent / reload scene) and `DamageDealer` (damage amount + per-target cooldown) are child-node components, instanced as direct children of a body's scene root and configured per instance in the `.tscn`. `health_changed(current, max)` fires on every change and once on `_ready`, so UI (HUD, character sheet) can bind to it as the single source of truth; the player's receiver is in group `"player_health"` for discovery. The convention: the *owner* detects contact (projectile impact callback, spider slide collisions) and calls `dealer.try_deal(body)`; the dealer finds the body's receiver via `DamageReceiver.find_in(body)` (direct-children scan, so receivers must sit at the body root). Both are optional everywhere â€” scripts look them up with `get_node_or_null` and skip damage if absent, so a scene without components still works. Current wiring: player receiver (100 HP, reload scene on death), spider dealer (touch, 10 dmg / 1s cooldown) + receiver (50 HP), barrel receiver (20 HP), Fireball/Ice Bolt projectile dealers (25/15 dmg). There is still no loot-drop system; death is queue_free (or scene reload for the player).

### Player (`Scenes/brackeys-proto-controller-main/`)
The player started as the third-party Brackeys `ProtoController` (`CharacterBody3D`, group `"player"`, `class_name PlayerController`) but is now heavily adapted: it is the **"context"** for a state machine (below), keeping only what's true in *every* state â€” mouse-look + capture, exported config (speeds/input-action names), node refs, the sprint-stamina hook (`_try_pay_sprint`) â€” plus the movement *primitives* the states call (`apply_gravity`, `wish_direction`, `apply_horizontal_velocity`, `resolve_move_speed`, `freefly_move`). It no longer has a locomotion `_physics_process`; states own movement. Keep further gameplay systems hooked in via autoloads/components/child-nodes rather than baked into this file (`SpellCaster`, `PlayerInteractor`, `PlayerAnimator`, `StateMachine` are all child nodes). It self-manages mouse capture (left-click captures, Escape releases) in `_unhandled_input`. Anything that opens a full-screen UI must consume input at the `Control` layer (`mouse_filter = STOP` on a full-rect backdrop) so clicks don't fall through and re-trigger capture.

### State machine (`Scripts/StateMachine/`, player states in `Scripts/Player/States/`)
A **generic, reusable FSM**: `StateMachine` (a `Node`) holds `FsmState` children, forwards `update`/`physics_update`/`handle_input` to the active one, and switches when a state emits `transition_requested` (target = a sibling state's node name). It knows nothing about the player, so the spider/NPCs can reuse it. **NB the base class is `FsmState`, not `State`** â€” `State` collided with a local `enum State` in `skill_node_button.gd` and broke class resolution project-wide; keep generic names distinct. The `StateMachine` **defers its first `enter()`** (`call_deferred`) so states can resolve refs (player/animator via groups) after the whole tree is `_ready`.
Player states (`PlayerFsmState` base â†’ `GroundedState`/`JumpState`/`FallState`/`FreeflyState`/`DeadState`, authored as nodes under the player's `StateMachine`) own movement + pick the animation state. `PlayerFsmState` resolves `player`/`animator` lazily and handles the freefly toggle for all states. **Casting and hurt are NOT states** â€” they're animation overlays (see below), so you can cast/flinch while moving. **Death IS a state**: the player `DamageReceiver` is `death_behavior = NONE` and `PlayerController` routes its `died` â†’ `transition_to("Dead")`; `DeadState` plays `Death_A`, freezes control, and reloads the scene after `reload_delay` (2.5s) â€” progression survives via autoloads. (A hit while dead can't re-enter; `DeadState` ignores input.)

### Player animation (`Scripts/Player/player_animator.gd`)
`PlayerAnimator` (child node, group `"player_animator"`) is the **only thing that touches the AnimationPlayer/Tree** â€” states call `travel("Grounded"/"Jump"/"Fall")`, `set_locomotion(speed01)`, `play_hurt()`, `play_cast_shot()`, `play_land()`, `play_death()`. It drives an assigned **`AnimationTree`** (the intended path) OR falls back to `AnimationPlayer.play()` on mapped clips when no tree is assigned, so the character animates before the tree is authored. The **cast overlay is split**: held spells ramp `parameters/CastBlend/blend_amount` (sustained `Ranged_Magic_Spellcasting`) off `SpellCaster.is_casting()`; instant spells fire a `parameters/CastShot/request` OneShot (`Ranged_Magic_Shoot`) off the `SpellCaster.spell_cast` signal (`spell.instant`). Landing fires `parameters/LandShot/request`. **Death bypasses the tree** â€” `play_death()` sets `animation_tree.active = false` and plays `Death_A` on the plain AnimationPlayer (so Death needs no movement-SM state); the scene reload spawns a fresh player with the tree active again. **AnimationTree contract (author in-editor, then assign to the animator):** the root is an `AnimationNodeBlendTree` (so the cast/hurt overlays can layer over movement); the movement `AnimationNodeStateMachine` is **nested inside it as a node named `Movement`**, with states `Grounded`/`Jump`/`Fall` (travelled by code). Nested nodes take the parent name as a path prefix, so the paths are: `parameters/Movement/playback` (state travel); `parameters/Movement/Grounded/blend_position` = `Idle_B`â†’`Walking_B`â†’`Running_B` blend space; `parameters/CastBlend/blend_amount` = a `Blend2` of the `Movement` output vs a looping `Ranged_Magic_Spellcasting`, filtered to the upper body (spine/arms); `parameters/HurtShot/request` = a `OneShot` of `Hit_A` on top (upper-body filtered). Two **optional** overlay one-shots the code also fires (add them when wanted â€” they no-op silently until they exist, see "Next steps"): `parameters/CastShot/request` (`Ranged_Magic_Shoot`, upper-body, for instant casts) and `parameters/LandShot/request` (`Jump_Land`, full-body, on landing). All these paths live in the `PARAM_*` constants in `player_animator.gd` â€” `AnimationTree.set()` on a wrong path fails **silently**, so keep the constants and the authored graph in sync. Clip libraries on the player: `Rig_Medium_General` / `_MovementBasic` / `_CombatRanged` (KayKit).

### Interaction (`Components/interactable.gd`, `Scripts/Player/player_interactor.gd`, `Scenes/UI/dialogue_screen.tscn`)
`Interactable` is a drop-in component (child node, `find_in(body)` direct-children scan like `DamageReceiver`) on any body that should respond to the `interact` action (F); the body must join group `"interactable"` so the raycast finds it. It exposes a `prompt` verb and emits `interacted(interactor)`, and if `dialogue_lines` (an `Array[String]`) are set it opens the dialogue screen â€” so "talk to an NPC" works with zero code, while the signal lets quests/doors/pickups hook their own behaviour in without subclassing. `PlayerInteractor` (a `Node` under the player's `Camera3D`, sibling of `SpellCaster`) raycasts from the crosshair each physics frame, gates on distance **from the player** (not the third-person camera), pushes the prompt to the HUD (`set_interact_prompt()`, a centered label below the crosshair), and calls `interact()` on press â€” all gated on `MOUSE_MODE_CAPTURED` so it's inert in menus. The **`DialogueUI` autoload** (`DialogueScreen` on the scene's `Screen` child, **found via group `"dialogue_screen"` â€” NOT `/root/DialogueUI`, which is the CanvasLayer**) is a bottom-panel read-only dialogue following all the menu-screen conventions (pause + mouse release, theme, backdrop, `process_mode = ALWAYS`, HUD-hide, mutually exclusive via `"menu_screen"`): `start(speaker, lines)` shows one line at a time, advancing on click / `ui_accept`, closing on the last line or `ui_cancel`. Built to grow into branching choices + NPC disposition later. The dock foreman (`NPCs/dock_foreman.tscn`) is killable (its `DamageReceiver`, the violence path) and talkable â€” and is now an inherited instance of the data-driven NPC base (see "Data-driven NPCs" above), with its lines living in `foreman_profile.tres`.

### UI (`Scripts/UI/`, `Scenes/UI/`)
The inventory/character screen is the first UI in the project and sets the conventions for future screens:
- Registered as an **autoload singleton** (`InventoryUI` in `project.godot`) rather than a node manually placed in every level, so it's available everywhere and isn't tied to `level.tscn`.
- Toggled by the `inventory` input action (Tab, already mapped in `project.godot` before this system existed). Opening sets `get_tree().paused = true` and `Input.mouse_mode = VISIBLE`; the screen node and everything under it must have `process_mode = PROCESS_MODE_ALWAYS` (set on the root `CanvasLayer`) or it stops receiving input the moment the tree pauses.
- `InventorySlot` (`Scenes/UI/inventory_slot.tscn`) is a single dual-purpose component used for both backpack grid cells and paper-doll equipment sockets (`is_equipment_slot` + `accepted_equip_slot` toggle the behavior) â€” not two separate slot classes. It owns its own drag-and-drop (`_get_drag_data`/`_can_drop_data`/`_drop_data`) and double-click quick-transfer, and notifies the screen via `get_tree().get_first_node_in_group("inventory_screen")` rather than a direct reference.
- `InventoryItem` (`Scripts/UI/item.gd`) is a plain `Resource` data shape with no backing item/save system yet. `MockItemDatabase` (`Scripts/UI/mock_item_database.gd`) is placeholder demo data standing in for that system â€” replace it, don't extend it, once real loot/persistence exists.
- Shared look-and-feel lives in one `Theme` resource (`Scenes/UI/inventory_theme.tres`, dark panels + brass borders) applied at the screen root so new UI should pull from it rather than re-declaring styleboxes per-scene. This file gets hand-edited/re-saved by the Godot editor directly (not just through Claude), so re-read it before assuming its contents rather than trusting an earlier diff.

The **journal screen** (`Scripts/UI/journal_screen.gd`, `JournalUI` autoload, toggled by `journal`/J) is the second full-screen menu and copies all of the above conventions (autoload, backdrop, theme, `process_mode = ALWAYS`, pause + mouse release, HUD-hide). It's a **read-only log**: a left `ItemList` of entry titles + a right `RichTextLabel` for the selected entry's body (`JournalEntry` data shape â€” same mock-data pattern as `MockItemDatabase`/`MockSpellbook`). `add_entry(JournalEntry)` appends one at runtime; `set_entries(Array[JournalEntry])` replaces the whole log. `MockJournal` is now only a **generic fallback** shown when a level provides no objectives of its own.

**Per-level objectives** (`Scripts/Level/level_objectives.gd`): each level scene contains a `LevelObjectives` node (a plain `Node`, edited in the inspector â€” the single place to change a level's goals). On load it (deferred) pushes its `objective_title`/`objective_text` to the HUD objective panel (`group "hud"` â†’ `set_objective`) and its entries to the journal (`group "journal_screen"` â†’ `set_entries`), **replacing** the previous level's â€” so `level.tscn` shows "Remove the Foreman" + dock lore and `level_2.tscn` shows "Breach the Keep". `objective_in_journal` (default true) auto-adds the objective as the first journal entry so it's written once; `journal_entries: Array[JournalEntry]` holds the level's extra lore/clues (authored as `type="Resource"` sub-resources with `script` = `journal_entry.gd`; the typed-array serialises as `Array[JournalEntry]([SubResource(...)])`). A level without a `LevelObjectives` node falls back to the HUD's neutral default + `MockJournal`.

**Win state / level completion** (`Scripts/Level/Objectives/`, `Scripts/UI/level_complete_screen.gd`): `LevelObjectives` is also the level's **completion tracker**. The win *condition* is expressed as drop-in **`LevelObjective` components** placed anywhere in the level, which self-register with the tracker via the `"level_objectives"` group (same drop-in convention as `DamageReceiver`/`Interactable`). Each has `description` (its checklist line), `optional` (tracked+shown but doesn't gate the win), and `xp_reward`. The level is won when every **non-optional** objective completes â€” an AND; there's deliberately no OR/either-or support yet, which is the natural next step (see the foreman note below). Four condition types ship, and **adding a new one is a new subclass, touching nothing else** â€” override `_setup()` to hook whatever signal decides completion, then call `complete()` (or `set_progress(current, total)` for counted goals, which renders as "(2/3)" and auto-completes):
- `KillTargetObjective` â€” a body's `DamageReceiver.died`. Point `target` at the body, or parent the node to it. *Don't* parent it to a `FREE_PARENT` target if avoidable (the node is freed the moment it completes â€” handled, since the tracker caches each checklist row, but pointing `target` at it is tidier).
- `DestroyGroupObjective` â€” every member of `group` (default `"enemy_spawner"`) present **at level start** is gone, watched via `tree_exited`. Snapshotted once, so enemies a spawner emits later don't count. An empty group at start `push_warning`s and completes immediately rather than soft-locking.
- `ReachAreaObjective` â€” the `"player"` group enters an `Area3D`. Parent the node to the Area3D (it uses its parent) or set `area`.
- `InteractObjective` â€” an `Interactable.interacted` fires. Upgrade path once dialogue gets branching choices: complete on a specific *outcome* instead â€” swap the hooked signal, nothing else changes.

On completion the tracker awards per-objective + `completion_xp` via `PlayerProgression.add_experience`, then opens the **`LevelCompleteUI` autoload** (`LevelCompleteScreen`, script on the `Screen` child â†’ found via group `"level_complete_screen"`, like `DialogueScreen`). It follows every menu convention but is **not toggleable** â€” dismissed only via Continue, which loads the tracker's exported `next_level` (empty = last level, Continue just closes). Progression survives the swap for free since it lives in autoloads. Two ordering details that matter: the screen is opened **deferred**, because `Interactable.interact()` emits `interacted` *before* opening its dialogue and would otherwise let dialogue pop over the win screen; and `close()` runs **before** `change_scene_to_packed` so the next level doesn't inherit a paused tree. `LevelCompleteUI` is registered before `InventoryUI`/`JournalUI` in reverse-input order, so while open it swallows their toggles (otherwise opening a menu over it would close it via the `menu_screen` mutual exclusion and strand the player in a finished level).

The HUD objective panel renders a **live checklist** (`set_objective_items(title, rows)`, rows = `{text, complete, optional}`) whenever a level has objective components, falling back to the plain `set_objective(title, text)` when it doesn't. Currently wired: `level.tscn` = kill the foreman (required) + hear him out (optional) â†’ `level_2.tscn`; `level_2.tscn` = destroy the 3 nests + reach the inner hall. **Note the foreman's fiction promises "bribery, persuasion, or misdirection" but only killing him wins** â€” talking is currently just flavour lines that resolve nothing, so it's the optional objective; making persuasion a real win needs OR-support plus branching dialogue outcomes.

**Full-screen menus are mutually exclusive** via the `"menu_screen"` group: both `InventoryScreen` and `JournalScreen` join it and call `_close_other_menus()` in `open()`, so opening one closes the other â€” without this, closing a second-opened menu would unpause the tree and recapture the mouse while the first was still visible.

### Level scene (`Scenes/level.tscn`)
Mostly hundreds of hand-placed `Floor`/`Wall` tile instances plus a few set-piece nodes (`ProtoController`, `SpiderWalker`, pirate ship, barrels). When editing this file directly as text, be careful with `unique_id` node attributes and don't try to read/edit the whole floor-tile block â€” it's bulk, not structure.

### Input map (`project.godot`)
Actions are pre-declared even when unused by any script yet (e.g. `interact` on F was unused until recently, `inventory` on Tab similarly) â€” check `[input]` in `project.godot` before assuming an action needs to be created for a new feature.

### Magic loadout system (Fireball/Ice Bolt, spellbook tab)
`SpellCaster.equip_spell(trigger_action, spell_scene)` / `unequip_slot(trigger_action)` dynamically add/remove a `Spell`-derived scene as a child so a hotbar slot's binding can change at runtime â€” used by the inventory's Spellbook tab. Every equippable ability (`TelekinesisGrab/Push`, `FireballSpell`, `IceboltSpell`) is authored as its own standalone `.tscn` under `Scripts/Magic/` (not inline nodes in `proto_controller.tscn`) specifically so it can be `instance=`'d both statically in a scene *and* dynamically via `equip_spell()`. **Spells are now gated behind the skill trees (below), so the `proto_controller.tscn` SpellCaster is authored EMPTY** â€” no default loadout; children are added at runtime by `equip_spell()` when the player drags an unlocked spell onto a slot. `ProjectileSpell`/`Projectile` are generic (spawn-and-fly-and-impact) scripts parametrized per spell via scene config (mesh/particle colors, speed, impact force) rather than duplicated per spell â€” adding a third projectile spell means authoring a new `Projectile` scene, not new code. `SpellDefinition`/`MockSpellbook` mirror the `InventoryItem`/`MockItemDatabase` mock-data pattern.

### Magic skill trees (`Scripts/Progression/skill_*.gd`, Skills tab)
Per-school skill trees **gate every spell**: a spell only shows in the spellbook palette (and is only equippable) once its `UNLOCK_SPELL` node is unlocked. **`SkillSystem`** (autoload, registered right after `PlayerProgression` because it spends that system's `skill_points`; persists across scene reloads) owns the unlocked-node set + unlock rules (`can_unlock`/`unlock`/`is_unlocked`/`prerequisites_met`) and the effect readouts consumers use: `unlocked_spell_ids()` â€” the spellbook filters its palette by this, **fully wired** â€” plus `passive_total(key)` and `has_upgrade(id)`, which are **deliberately just scaffolding**: the payloads are stored and queryable but nothing reads them yet (flesh out per-spell passive/upgrade hooks in a later pass). Spending flows through new `PlayerProgression.spend_skill_point()`/`grant_skill_points()` so `skill_points` stays single-sourced (the HUD/character-sheet readouts bind to `points_changed` unchanged); `SkillSystem.reset()` is a working respec that refunds, with no UI yet.

Data is the mock-data pattern again: `SkillNode`/`SkillSchool` are `Resource`s authored in `MockSkillTrees` (replace with real `.tres` once a save/content pipeline exists). Nodes form a **DAG** â€” `prerequisites: Array[String]`, so branches can converge â€” and each carries a `grid_position` (col,row) that drives both layout and where connector lines are drawn. A node does exactly one thing via its `effect` enum (`UNLOCK_SPELL`/`PASSIVE`/`UPGRADE`). **Expansion is data-only**: add a *school* by appending to `MockSkillTrees.schools()` (the switcher + view iterate whatever exists); deepen a *tree* by appending `SkillNode`s with prerequisites â€” neither touches UI code. `UNLOCK_SPELL.unlock_spell_id` must match a `SpellDefinition` id in `MockSpellbook`.

UI: a **Skills tab** in the character screen (third tab). `SkillTreeView` (an instanced scene â†’ looked up by `$path` per the convention below) lays out a `SkillNodeButton` per node by `grid_position` and draws prereq connector lines in `_draw()` (so they sit behind the buttons); it owns unlock-on-click and re-renders on `SkillSystem.tree_changed`. `inventory_screen.gd` builds the per-school switcher buttons and drives the shared `ItemTooltip` off the view's hover signals; the spellbook palette also rebuilds on `tree_changed`. `PlayerProgression.starting_skill_points` (default 3) seeds a fresh game so a root spell or two can be unlocked immediately despite full gating â€” set to 0 for a start where nothing is castable until you level.

### `inventory_screen.tscn` convention: `$path` for instanced nodes, `%unique_name` for plain nodes
`InventorySlot`/`SpellSlot`/`ItemTooltip` nodes (placed via `instance=ExtResource(...)`) are looked up by explicit `$relative/path/To/Node` (see the `_CHAR_PATH`/`_SPELL_PATH` constants in `inventory_screen.gd`), not `%unique_name`. Plain built-in node types (`Button`, `Label`, `GridContainer`, etc.) use `%unique_name` as normal. This was originally adopted because an earlier session saw a `WARNING: Parent path '...' has vanished when instantiating` cascade and attributed it to `unique_name_in_owner` on instanced nodes corrupting `PackedScene::instantiate()`. **That diagnosis turned out to be wrong** â€” a later occurrence of the exact same warning (2026-07-17) was empirically bisected (isolated repro scenes, stripping `unique_name_in_owner` project-wide, de-duplicating node names â€” none of it helped) down to a mundane stale `parent=` NodePath: when a `Layout` `VBoxContainer` was inserted as a new wrapper around `Header`/`Tabs`, every descendant of `Tabs` kept the old pre-`Layout` parent path and was never updated, so Godot correctly reported the path as not found. **If you see this warning again: check `parent=` attributes against actual node nesting first** â€” grep for the container's own `parent=` line and confirm every one of its descendants' paths actually contains it â€” before suspecting `unique_name_in_owner` or any other engine-level explanation. The `$path`/`%unique_name` split above is kept as-is since it works and isn't worth re-churning, not because the original bug theory was confirmed.

Separately: don't hand-write `uid="uid://..."` attributes when authoring new `.tscn`/`.tres` files by hand (outside the editor) â€” a fabricated UID isn't registered anywhere and risks colliding with one Godot generates for something else during a full reimport. Omit `uid=` and reference by `path=` only; Godot fills in a real UID the next time the file is saved through the actual editor.
